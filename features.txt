1. Что такое «хороший» и «плохой» узел в Kumquad?
Тип	Поведение	Примеры
Хороший	Депозит получен → BTC отправлен пользователю в разумное время. Редирект принят → выполнен. Reserve честно объявлен.	Стабильный обработчик.
Плохой (мошенник)	Получил USDT (прямой депозит или редирект), но BTC не отправил. Резерв завышен, но реального BTC нет. Редирект отклоняет без причины.	Скамер.
Ненадёжный	Часто оффлайн, медленные ответы, резерв внезапно падает.	Нестабильный оператор.

Мошенник может:

    Принять USDT на свой адрес, а потом исчезнуть.

    Создать поддельный редирект от имени другого узла (амбициозно — нужно подписывать сообщения).

    Завышать резерв, чтобы привлекать редиректы, но не иметь BTC.

2. Общая архитектура реестра с репутацией

Каждый узел хранит локальную таблицу репутации для всех известных пиров:
rust

struct ReputationEntry {
    peer_id: String,
    score: f64,                 // -1.0 .. +1.0 или 0..100
    successful_redirects: u64,   // этот узел успешно выполнил редирект для нас
    failed_redirects: u64,
    accepted_redirects: u64,      // сколько раз он согласился на наш редирект
    avg_response_ms: u32,
    last_seen: u64,
    last_positive_event: u64,
    last_negative_event: u64,
    balance_trust: f64,          // насколько верим его заявленному резерву
    // optional: подписанные доказательства
}

Обмен репутацией происходит через gossip (как обсуждали ранее) — узлы обмениваются не только NodeInfo, но и сводками репутации (ReputationMessage). При слиянии репутации нельзя слепо доверять чужому счёту — нужна защита от накрутки.
3. Механизм редиректа с подтверждением

Текущий протокол редиректа (из спецификации) не гарантирует, что целевой узел реально отправит BTC. Доработаем:

    Узел A (получил депозит, мало резерва) шлёт на узел B запрос:
    json

    POST /api/p2p/redirect
    {
      "nonce": 12345,
      "deposit_txid": "0x...",
      "usdt_amount": 20.0,
      "chain": "tron",
      "user_btc_address": "bc1q...",
      "signature": "A.sign(nonce + deposit_txid + amount + chain + user_addr)"
    }

    Узел B проверяет подпись (чтобы убедиться, что запрос действительно от A). Проверяет свой резерв. Если согласен:

        Отправляет ответ { "accepted": true, "commitment": B.sign(accepted) }.

        Обещает в течение N минут отправить BTC на адрес пользователя.

    Узел B отправляет BTC (автоматически). После отправки генерирует квитанцию — подписанное сообщение с btc_txid и суммой.

    Узел A получает квитанцию (через колбэк или периодический опрос) и обновляет репутацию B: успех.

    Если квитанция не пришла через timeout, A помечает редирект как failed, снижает репутацию B и может распространить жалобу (Complaint) через gossip.

4. Криптографические доказательства для репутации

Чтобы злоумышленник не клеветал на хорошие узлы, каждое утверждение должно быть подписано:

    Успешный редирект → квитанция от B (содержит txid, сумму, адрес получателя). A может предъявить квитанцию другим узлам как доказательство надёжности B.

    Неудачный редирект → A может опубликовать свой запрос и отсутствие ответа (с таймстампом). Но это слабое доказательство. Лучше: A и B обмениваются подписанными сообщениями на каждом шаге.

Реестр хороших/плохих узлов может поддерживаться как DHT с отзывами (как в системах репутации eBay, но децентрализовано). Однако в Kumquad достаточно локальной репутации + gossip: каждый узел сам решает, кому доверять. Для повышения точности можно использовать кросс-валидацию: когда A рассказывает о B, C может проверить, действительно ли B имеет высокую репутацию у других.
5. Gossip для распространения репутации

При каждом обмене узлы посылают:
rust

struct GossipMessage {
    sender: NodeInfo,
    reputation_updates: Vec<ReputationUpdate>,   // только дельта, не вся таблица
    peers: Vec<NodeInfo>,
}

struct ReputationUpdate {
    peer_id: String,
    delta_score: f64,          // +0.1 за успешный редирект, -0.5 за провал
    proof: Option<SignedProof>, // например, квитанция или Complaint
}

При получении ReputationUpdate узел проверяет подпись доказательства (если есть) и применяет изменение к своей локальной таблице. Если доказательство отсутствует (просто слух), он может игнорировать или учитывать с меньшим весом.

Чтобы избежать лавины ложных обвинений, каждое отрицательное обновление должно сопровождаться подписанным Complaint, содержащим:

    ID редиректа / депозита.

    Подписанный запрос от инициатора.

    Доказательство истечения времени (например, блокчейн-хеш времени).

Это сложно, но реализуемо. Для v1 можно ограничиться только положительными доказательствами (квитанции) и локальным трекингом неудач без глобального распространения.
6. Алгоритм выбора «хорошего» узла для редиректа

При необходимости редиректа узел A:

    Берёт из реестра всех узлов с status == "online" и заявленным резервом >= нужному.

    Фильтрует по своей локальной репутации: score >= MIN_SCORE (например, 0.7).

    Сортирует по комбинированному метрику: score * (1 - avg_response_ms/10000) или просто по score.

    Выбирает лучшего и отправляет запрос.

Если ни одного подходящего нет — редирект не делается, депозит остаётся на ожидании ручной обработки оператором A.
7. Критерии «плохой узел» (что снижает репутацию)
Событие	Штраф
Отказ принять редирект с причинами insufficient_reserve	0 (честный отказ)
Отказ без причины или таймаут	-0.2
Принял редирект, но не отправил BTC (квитанция не пришла)	-0.8 и занесение в локальный чёрный список
Отправка BTC с большой задержкой (>30 мин)	-0.1
Заявленный резерв часто не соответствует реальному (проверяется через /api/p2p/reserve в разное время)	-0.05 за несоответствие
Подписанное доказательство мошенничества от другого узла	-1.0 (мгновенная изоляция)
Оффлайн > 1 часа	score постепенно снижается до 0

Бонусы:

    Успешный редирект (квитанция получена) → +0.1

    Постоянно высокий резерв и малая задержка → небольшой ежечасный бонус

8. Реализация локальной репутации (без глобального распространения)

Самый простой вариант, который можно сделать уже сейчас:

    Узел ведёт свою таблицу peer_id -> (success_count, failure_count).

    Вычисляет доверие как (successes+1)/(successes+failures+2).

    Использует её для выбора узлов при редиректе.

    При gossip обменивается только NodeInfo, а репутацию не распространяет.

    Это не позволяет быстро узнать о плохом узле, но защищает от накрутки. Для небольших сетей (до 50 узлов) достаточно.

9. Будущее: децентрализованная система репутации с подписанными доказательствами

Можно реализовать реестр хороших узлов как DHT с рейтингом, похожий на токен доверия:

    Узел, успешно выполнивший редирект, получает подписанную квитанцию.

    Квитанция может быть предъявлена любому другому узлу как доказательство надёжности.

    Другие узлы могут повысить свою локальную репутацию для этого узла на основе предъявленной квитанции (даже если они не были участниками сделки).

    Для защиты от подделки квитанция содержит подпись получателя BTC? сложно.

На начальном этапе достаточно локальной репутации + чёрных списков, распространяемых через gossip как слухи (без криптодоказательств). Для серьёзного продакшена потребуются доказательства.
10. План интеграции в Kumquad (поэтапно)

    Базовая локальная репутация (v0.4):

        Добавить PeerRegistry с полями successful_redirects, failed_redirects.

        При редиректе и получении квитанции обновлять счётчики.

        Выбор узла по score (на основе истории).

        Gossip только для NodeInfo, без репутации.

    Распространение жалоб (v0.5):

        Добавить в gossip Complaint сообщения (подписанные инициатором редиректа).

        Узлы, получившие жалобу, могут снизить репутацию без доказательств, но с ограничением (не более 10 жалоб за день на одного пира).


1. Требования к gossip и реестру
Функция	Назначение
PeerRegistry	Хранит NodeInfo (peer_id, http_addr, chains, btc_reserve, fee_usd, last_seen, status). Конвергентная репликация.
Gossip	Периодически обмениваться сводками (с вероятностью дельта-обновлений), быстро распространять изменения резерва или статуса.
Fault detection	Узлы, которые не отвечают на ping / резерв не обновлялся > N минут, помечаются stale и удаляются.
Лёгкий вес	Без центрального координатора, только P2P.
2. Выбор протокола

Варианты:

    GossipSub (libp2p) — мощный, но тяжёлый, требует mesh-поддержки.

    Epidemic Broadcast / Push-Pull gossip (как в BitTorrent DHT, Serf) — проще, можно реализовать поверх Kademlia или отдельным UDP-каналом.

    Periodic Peer Exchange (как в Bitcoin) — каждый узел обменивается случайной выборкой пиров.

Для Kumquad предлагаю гибрид:

    Реестр хранится как Local K/V (sled или HashMap) + реплицируется через gossip-обмен (push-pull).

    DHT используется только для первичного поиска: узел публикует свой NodeInfo по ключу /kum4/node/{peer_id}. Другие узлы находят его через DHT.

    Для живучести используется поверхностный ping-протокол и обмен сводками (membership list).

3. Реализация реестра (PeerRegistry)
rust

// p2p/registry.rs
use std::collections::HashMap;
use std::sync::Arc;
use tokio::sync::RwLock;
use std::time::{Duration, Instant};

#[derive(Clone, Serialize, Deserialize, Debug)]
pub struct NodeInfo {
    pub peer_id: String,
    pub http_addr: String,        // "http://1.2.3.4:8080" или onion
    pub btc_reserve: f64,
    pub fee_usd: f64,
    pub chains: Vec<String>,
    pub status: String,           // "online", "busy", "offline"
    pub last_seen: u64,           // Unix timestamp
    pub version: String,
}

impl NodeInfo {
    pub fn is_stale(&self, now: u64, timeout_secs: u64) -> bool {
        now - self.last_seen > timeout_secs
    }
}

pub struct PeerRegistry {
    peers: Arc<RwLock<HashMap<String, NodeInfo>>>,
    stale_timeout: Duration,  // e.g., 3600 sec
}

impl PeerRegistry {
    pub fn new(stale_timeout_secs: u64) -> Self {
        Self {
            peers: Arc::new(RwLock::new(HashMap::new())),
            stale_timeout: Duration::from_secs(stale_timeout_secs),
        }
    }

    pub async fn update(&self, info: NodeInfo) {
        let mut map = self.peers.write().await;
        map.insert(info.peer_id.clone(), info);
    }

    pub async fn get(&self, peer_id: &str) -> Option<NodeInfo> {
        self.peers.read().await.get(peer_id).cloned()
    }

    pub async fn remove_stale(&self, now: u64) -> Vec<String> {
        let mut map = self.peers.write().await;
        let stale: Vec<_> = map
            .iter()
            .filter(|(_, v)| v.is_stale(now, self.stale_timeout.as_secs()))
            .map(|(k, _)| k.clone())
            .collect();
        for id in &stale {
            map.remove(id);
        }
        stale
    }

    pub async fn all_active(&self, now: u64) -> Vec<NodeInfo> {
        self.peers
            .read()
            .await
            .values()
            .filter(|v| !v.is_stale(now, self.stale_timeout.as_secs()))
            .cloned()
            .collect()
    }
}

4. Gossip протокол (push-pull)
4.1 Идея

    Каждый узел периодически (например, раз в 60 секунд) выбирает K случайных активных пиров из своего реестра.

    Запрашивает у них сводку (их реестр) — протокол GossipPull.

    Отправляет им свою сводку — GossipPush.

    Слияние происходит по принципу «последняя запись победит» + более свежий last_seen выигрывает.

    Используется UDP для экономии, но можно и TCP (libp2p stream).

4.2 Формат сообщений
rust

#[derive(Serialize, Deserialize)]
pub struct GossipMessage {
    pub sender: NodeInfo,
    pub peers: Vec<NodeInfo>,   // subset or full registry
}

4.4 Gossip задача на tokio
rust

async fn gossip_task(registry: Arc<PeerRegistry>, swarm: Option<Swarm>) {
    let mut interval = tokio::time::interval(Duration::from_secs(60));
    let rng = rand::thread_rng();

    loop {
        interval.tick().await;
        let now = SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap()
            .as_secs();

        // получить список активных пиров
        let peers = registry.all_active(now).await;
        if peers.is_empty() {
            continue;
        }

        // выбрать случайных 3
        let num = std::cmp::min(3, peers.len());
        let selected = peers.choose_multiple(&mut rng, num);

        for target in selected {
            // сформировать сводку: можно всю (если узлов до 100), или дельту.
            let my_info = NodeInfo {
                peer_id: local_peer_id(),
                btc_reserve: get_current_reserve(),
                last_seen: now,
                // ...
            };
            let msg = GossipMessage {
                sender: my_info.clone(),
                peers: registry.all_active(now).await, // full sync for simplicity
            };

            // отправить target на его gossip порт (UDP или libp2p)
            send_gossip(target, &msg).await;

            // получить ответ и смержить
            if let Ok(reply) = recv_gossip(target, timeout).await {
                merge_gossip(&registry, reply).await;
            }
        }

        // удалить stale записи
        registry.remove_stale(now).await;
    }
}

4.5 Слияние (merge)
rust

async fn merge_gossip(registry: &PeerRegistry, msg: GossipMessage) {
    let now = current_timestamp();
    let mut to_update = vec![msg.sender];
    to_update.extend(msg.peers);
    for info in to_update {
        // только если информация свежее текущей
        let existing = registry.get(&info.peer_id).await;
        match existing {
            Some(ex) if ex.last_seen >= info.last_seen => continue,
            _ => registry.update(info).await,
        }
    }
}

5. Интеграция с DHT Kademlia

    DHT используется для начального поиска пиров и для публикации собственного NodeInfo.

    Каждый узел пере-объявляет свою запись /kum4/node/{peer_id} каждые 15 минут.

    При старте узел загружает из DHT записи других узлов и добавляет их в реестр.

    Gossip затем поддерживает реестр в актуальном состоянии.

Пример загрузки из DHT при старте:
rust

async fn bootstrap_from_dht(&mut self) {
    // найти все записи по ключевому префиксу /kum4/node/
    for peer_id in self.dht.find_providers(Key::new("/kum4/node/")) {
        if let Some(info) = self.dht.get_record(&format!("/kum4/node/{}", peer_id)).await {
            self.registry.update(info).await;
        }
    }
}

6. Обнаружение сбоев (failure detection)

    Каждый узел периодически (раз в 2 минуты) пингует случайные 5 узлов из реестра.

    Пинг — это простой HTTP GET /api/p2p/ping на их http_addr.

    Если ответа нет 3 раза подряд, узел помечается как status: "offline" и его last_seen не обновляется.

    Через stale_timeout (1 час) узел удаляется из реестра.

Дополнительно: можно использовать gossip as liveness — если в последнем обмене узел не прислал свой NodeInfo, другие узлы перестают его распространять.
7. Протокол редиректа (уже есть в spec)

Редирект на другой узел при нехватке резерва:

    Узел A получает депозит.

    Проверяет свой резерв: если недостаточно, ищет в реестре peer с btc_reserve >= нужное.

    Отправляет POST /api/p2p/redirect на http_addr этого peer с { usdt_amount, chain, user_btc_address, deposit_txid }.

    Peer B возвращает accepted: true и обрабатывает депозит как свой (B отправляет BTC пользователю, B забирает USDT себе как доход).

    Для компенсации между операторами — вне этой спецификации (отдельные договорённости).

8. Простое решение без отдельного gossip (только DHT)

Можно вообще не реализовывать сложный gossip, если число узлов невелико (≤20). Достаточно:

    Регулярно (раз в 10 минут) пере-публиковать свой NodeInfo в DHT.

    При запросе реестра (/api/p2p/nodes) — делать lookup в DHT по ключу /kum4/node/ и возвращать все найденные записи. Но это будет медленно и нестабильно (DHT не гарантирует полноту).

Gossip нужен для надёжного и быстрого распространения.
9. Вывод: что реализовать в Kumquad v0.4

    PeerRegistry с last_seen и автоматической чисткой.

    Gossip task на UDP (или через libp2p) — push-pull раз в 60 секунд.

    Ping task для проверки живости (через HTTP /api/p2p/ping).

    DHT bootstrap для начального заполнения реестра.

    Протокол редиректа (уже описан в спецификации, надо допилить с проверкой резерва).
